Refine the current personal website for @po4yka / Nikita Pochaev by adding more interesting hacking, IoT, mobile-device, and neo-retro vibes — but in a tasteful, modern, high-design way.

Do not redesign from scratch.
Preserve the current foundation:
- minimal editorial layout
- typography-first structure
- restrained palette
- clean portfolio / resume / blog architecture
- calm, premium, developer-focused tone

Main goal:
Evolve the website into something that feels more distinctive and more “po4yka” by introducing subtle influences from:
- hacking culture
- embedded / IoT systems
- mobile devices
- firmware / tooling / release engineering
- neo-retro digital aesthetics
- restrained glitch / signal distortion details

Important:
This must NOT become a cheesy hacker theme, a cyberpunk cliché, or a noisy terminal gimmick.
Do NOT turn it into:
- a fake dashboard
- a dark neon security landing page
- a Matrix parody
- a startup landing page with “cyber” decorations
- a retro cosplay website

The result should feel like:
- a real mobile engineer’s personal website
- minimal but more characterful
- editorial but more technical
- clean but slightly experimental
- premium but not polished into generic AI sameness
- calm with controlled tension
- modern with selective nostalgic signal artifacts

Core creative direction:
Think of this as:
“high-taste developer portfolio + firmware lab + mobile device workshop + subtle signal corruption”
not
“hacker movie interface”

Visual mood:
- precision instrument
- embedded systems notebook
- mobile device teardown / diagnostics
- terminal-adjacent but not terminal-dominated
- schematic / trace / signal / debug overlays
- archival digital texture
- restrained screen artifact energy
- slight glitch tension inside an otherwise disciplined layout

Design instructions:

1. Overall art direction
Add more identity through selective neo-retro technical cues:
- signal lines
- thin measurement marks
- subtle device/schematic framing
- firmware-like micro labels
- small system annotations
- understated debug-style metadata
- controlled glitch accents

Everything must remain refined and sparse.
The site should still feel mostly clean and readable from a distance.

2. Color and material system
Keep the overall light, restrained foundation, but introduce one more distinctive technical accent direction.

Preferred palette direction:
- warm light gray / paper gray / soft concrete background
- carbon / charcoal typography
- one controlled technical accent:
  - phosphor green, or
  - muted cyan, or
  - oxidized teal, or
  - amber signal orange

Optional:
Introduce extremely subtle secondary artifact tones in tiny details only, such as:
- desaturated mint
- dusty aqua
- faded CRT blue
- pale warning amber

Do not use bright neon cyberpunk colors.
Do not use heavy gradients.
Do not make the whole page dark unless it is a very controlled alt-theme concept.

3. Typography
Keep strong modern editorial typography, but add more technical personality.

Use:
- one strong sans-serif for primary content
- one mono for metadata, coordinates, dates, tags, and technical labels

Optional:
Add a very subtle pixel / LCD / terminal influence only in tiny doses:
- section indices
- system markers
- status labels
- project metadata
- micro headings

Do not make the whole site mono.
Do not make it look like an old terminal emulator.

4. Hero section
Keep the hero clean, but give it more technical identity.

Possible additions:
- a subtle status line under or near the handle, like a build / channel / platform indicator
- tiny coordinate-like labels
- soft alignment marks or calibration ticks
- a very restrained signal artifact near the heading or hero edge
- a small mobile-device / diagnostics-inspired visual fragment
- a thin scanning or sweep detail used once, not constantly

The hero should still feel calm and premium.
No giant illustrations.
No fake HUD overlays covering the content.

5. Project section
This is the best place to introduce mobile, IoT, and hacking vibes.

Redesign project previews so they feel like:
- app/device inspection panels
- firmware test benches
- mobile hardware diagnostics cards
- embedded system showcase modules
- developer lab snapshots

Possible visual ideas:
- phone/device frame fragments
- diagnostic panel overlays
- tiny connection-path or PCB-trace-like lines
- schematic corner markers
- compact platform status blocks
- micro labels like BUILD, TARGET, PLATFORM, CHANNEL, RELEASE, STATE
- subtle wireframe / sensor / module style annotations
- cropped mobile UI inside precise device-like frames
- small signal artifacts on image edges
- careful use of scanline or RGB split only inside preview media

Do not make previews busy.
Do not fake entire dashboards.
Each preview should still read as a premium project showcase first.

6. Blog section
Give the blog more “engineering notes / lab notebook / field notes” energy.

Possible refinements:
- article metadata styled like system annotations
- dates and categories treated like release / log entries
- subtle index numbering
- slightly more technical dividers
- featured post framed like a highlighted signal capture / note card / dev memo

The blog should still feel editorial and highly readable.
Avoid over-theming reading UI.

7. Interesting details to add
Introduce small signature details across the site such as:
- section numbers like 01 / 02 / 03
- tiny mono coordinates
- build/version markers
- release channel labels
- hardware-annotation style arrows
- calibration ticks near large blocks
- faint scanline or phosphor texture in isolated zones
- subtle pixel-grid texture in media or hover states
- controlled RGB separation on hover for key words or preview edges
- tiny status LEDs / signal dots used very sparingly
- microcopy like:
  - channel: stable
  - target: android / ios
  - build: kmp
  - mode: notes / projects / archive

These should be tasteful and sparse, not decorative clutter.

8. Glitch / distortion direction
Add glitch-inspired details, but keep them sophisticated and minimal.

Good glitch ideas:
- 1-frame style RGB split effect on hover
- subtle horizontal signal tear on featured project or hero asset
- tiny interference lines in media previews
- brief text duplication / shadow offset on interaction
- controlled distortion on separators or preview masks
- scanline sweep or sync-line effect used occasionally
- ghosted duplicate typography layer with very low opacity
- blend-like overlays for select visuals only

Important:
Glitch must be:
- occasional
- subtle
- optional
- art-directed
- never constant
- never full-screen
- never aggressive
- never flashing

Use glitch like seasoning, not the whole meal.

9. Motion and interaction
Motion should reflect signal, diagnostics, and device behavior rather than generic website animation.

Suggested motion ideas:
- subtle scan / sweep over preview media
- divider line draw that feels like a signal trace
- title hover with restrained RGB split or ghost offset
- tiny cursor-like or caret-like active navigation indicator
- project preview hover with micro parallax between frame and content
- metadata activating like system labels
- section labels revealing like log markers
- blog rows reacting like technical note entries
- gentle pulse or state change on tiny accent dots

Avoid:
- cinematic intros
- large parallax fields
- constant floating movement
- strong distortion loops
- fake hacking animations
- overcomplicated WebGL scenes
- motion that harms readability

10. Accessibility and restraint
All visual and motion additions must stay accessible and mature.

Rules:
- preserve excellent readability and contrast
- keep content first
- reduce or remove non-essential motion when reduced-motion is enabled
- avoid frequent flashing, harsh jitter, or intense glitch behavior
- ensure the website still looks strong if all effects are disabled

11. Anti-cliché rules
Never use:
- green-on-black terminal for the full site
- matrix rain
- skull / anonymous / red-team stereotypes
- padlock/security hero art
- fake code rain backgrounds
- random circuit-board wallpaper
- glowing holographic cards
- loud cyberpunk neon gradients
- bulky “techno” frames around everything
- fake command center UI
- overdesigned dark hacker dashboard tropes

12. Final target outcome
The final result should feel like a personal website for a mobile engineer who is into:
- Android and iOS
- Kotlin Multiplatform
- MobileOps
- tooling
- embedded / IoT curiosity
- digital artifacts
- neo-retro technical aesthetics

It should feel:
- more distinctive
- more authored
- more experimental
- more technical
- more memorable

But still:
- minimal
- readable
- premium
- structured
- believable
- useful as a real portfolio / resume / blog

The final design should look like a calm, art-directed developer website with subtle firmware-lab / mobile-diagnostics / neo-retro signal vibes — not a themed gimmick.
